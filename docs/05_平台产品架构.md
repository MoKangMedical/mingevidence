# 明证平台产品架构

## 目标

把 `明证` 从“可演示的中国版 OpenEvidence 原型”升级成“医生和医院可以直接使用的平台”。

这意味着产品不能停留在：

- 搜索页
- 单轮问答
- 文档检索

而必须升级成：

- 面向医生的证据入口
- 面向患者的病例工作台
- 面向药师/专科的药品与标签中心
- 面向 MDT 的中国人群预警面板
- 面向医院的治理后台

## 参考依据

这轮平台升级参考了当前可验证的官方产品和标准资料，核心启发如下：

### 1. OpenEvidence

- 面向医生而不是泛患者
- 以结构化答案和文献引用为核心
- 强调临床可直接使用，而不是纯聊天体验

### 2. UpToDate / DynaMed / ClinicalKey AI

- 共同特点不是“模型更强”，而是：
  - 证据层级清晰
  - 问题到答案路径短
  - 能直接嵌入临床工作流
  - 对药品、诊疗路径和文献形成统一入口

### 3. SMART on FHIR / EHR 集成

- 真正的医院级产品最终需要进入：
  - 患者上下文
  - 医生单点登录
  - 检验、诊断、用药和病程数据

### 4. PubMed / openFDA / NMPA

- `明证` 的知识底座必须持续依赖正式源和标准化元数据
- 不能把模型输出当成知识源本身

## 平台信息架构

## 一层：医生工作区

### A. 临床搜索

作用：

- 面向“我现在有个问题”的即时场景

必须输出：

- 结构化结论
- 中国指南位阶
- 药品监管边界
- 关键研究证据
- 风险提示

核心要求：

- 3 秒内给出第一屏
- 不要求用户先学会系统

### B. 病例工作台

作用：

- 面向“我在管理一个真实患者”的连续场景

核心对象：

- 单患者会话
- 问题拆解
- 多轮方案比较
- 指标与病程追踪

核心要求：

- 支持保存
- 支持追问
- 支持团队讨论

### C. 药品与标签中心

作用：

- 面向适应症、剂量、禁忌、不良反应和标签差异核查

核心视图：

- NMPA 标签
- FDA 标签
- 说明书原文
- 剂量/禁忌摘要
- 中国可及性与医保说明

### D. 中国人群预警面板

作用：

- 面向风险预测、复发预警、不良事件监测和随访

核心视图：

- 风险热图
- 病程时间轴
- 中国人群证据强度
- 监测节点提醒

## 二层：机构工作区

### A. 机构治理后台

面向：

- 科室负责人
- 医务部门
- 药学部门
- 信息部门

核心功能：

- 用户和角色管理
- 高风险问题审计
- 反馈与纠错管理
- 内容版本追踪
- 院内知识接入

### B. 内容运营后台

核心功能：

- 正式源同步监控
- 失败源告警
- 标签/指南版本对比
- 发布审批

## 平台分层架构

### 1. 体验层

- 搜索页
- 病例工作台
- 药品中心
- 预警面板
- 管理后台

原则：

- 入口按场景组织，不按技术功能组织

### 2. 应用服务层

- 医生认证
- 病例会话
- 收藏与历史
- 反馈与协作
- 通知与任务
- 审计日志

原则：

- 让搜索结果变成可持续使用的工作资产

### 3. 证据引擎层

- 问题理解与分流
- 关键词检索
- chunk rerank
- 指南/标签优先级控制
- 药品标签差异对照
- 风险问题拒答

原则：

- “中国正式证据优先，国际研究补充，监管边界单独显式”

### 4. 知识底座层

- 中国指南
- NMPA 文件
- FDA 标签
- PubMed 研究
- 中国人群分层信号

原则：

- 统一 schema
- 统一版本
- 统一引用输出

### 5. 治理与集成层

- 权限与审计
- 医疗风险控制
- 版本回滚
- EHR / SMART on FHIR
- 院内数据接入

原则：

- 平台先可信，再集成，再规模化

## 用户体验原则

## 1. 默认从问题开始

不要让医生先选择十几个筛选器。

首屏应该先允许：

- 直接提问
- 直接看答案
- 再逐步展开证据和边界

## 2. 默认结构化

回答不能是一大段话。

必须拆成：

- 一句话结论
- 适用对象
- 不适用对象
- 证据来源
- 下一步建议

## 3. 默认可视化

关键复杂场景必须可视化，而不是纯文本化：

- 病例要用时间轴
- 风险要用热图区间
- 方案比较要用矩阵
- 药品差异要用对照表

## 4. 默认保守

高风险问题要优先：

- 拒答
- 降级
- 提示人工复核

而不是勉强生成“看起来完整”的答案

## 版本路线

## V1

- 搜索
- DeepConsult
- 标签中心基础版
- 进展看板

## V1.5

- 病例工作台
- 中国人群预警视图
- 收藏、历史、反馈闭环

## V2

- 机构治理后台
- 医院试点版
- EHR / SMART on FHIR 嵌入

## 对当前仓库的直接影响

这套架构意味着接下来代码不应只围绕 `/search` 和 `/api/search` 扩展，而应新增：

- `病例工作台` 页面与数据模型
- `药品与标签中心` 页面与对照视图
- `中国人群预警面板` 页面
- `机构后台` 页面与角色权限
- `病例 / 搜索 / 药品 / 预警` 四类统一导航

也意味着数据层应新增：

- `patient_case`
- `case_note`
- `saved_search`
- `watchlist`
- `institution_role`
- `content_release`

## 结论

`明证` 下一阶段的核心，不是把搜索结果做得更像 ChatGPT，而是把它做成：

- 医生能每天打开
- 药师能核对边界
- MDT 能共同研判
- 医院能放心接入

的平台。
